Ciberseguridad en balizas V16 conectadas

Introducción + práctica (2h)

Objetivo: entender riesgo y validar controles defensivos.

¿Qué pasa cuando una baliza emite un aviso?

  • La baliza detecta incidencia.
  • Envía aviso al backend.
  • La plataforma decide: aceptar o rechazar.

Flujo extremo a extremo

  1. Dispositivo V16
  2. Red de comunicaciones
  3. Backend receptor
  4. Plataforma/servicios de tráfico

Activos críticos

  • Identidad del dispositivo
  • Integridad del aviso
  • Frescura temporal
  • Disponibilidad
  • Trazabilidad

Amenazas clave

  • Suplantación
  • Replay
  • Datos fuera de rango
  • Ráfaga / abuso de API

Prioridad defensiva

  • Alta: replay, suplantación, ráfaga
  • Media: timestamp/coords inválidas

Controles del backend

  • Esquema JSON estricto
  • security.sent_at (ventana temporal)
  • security.nonce (anti-replay)
  • Rate limiting por identidad
  • Logging con reason

Qué medimos en la práctica

  • Aceptación de inválidos (debe tender a 0)
  • Rechazo con motivo técnico
  • Trazabilidad para informe

Panel: /aginte-panela · JSON: /events/recent

Riesgo residual

  • Faltan controles de producción completos
  • Próximos pasos: firma, PKI, mTLS, SOC

Transición al laboratorio

  1. Aviso legítimo
  2. Replay e inválidos
  3. Observación panel/log
  4. Informe de evidencias

Solo sandbox local o red aislada.
